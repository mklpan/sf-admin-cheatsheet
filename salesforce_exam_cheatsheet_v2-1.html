Answer Strategies
1
Core Method
Eliminate First, Identify Second
Eliminating 3 wrong answers = finding the right one. Often easier than direct identification.
Wrong answers fail predictably: wrong object/context, exaggerated scope, impossible combinations.
You only need majority confidence — not certainty — to eliminate.
When stuck between 2, focus entirely on what's different between them — that's what the question is testing.
2
Red Flag Words
Absolute Language = Danger
Flag immediately: always never every all automatically only cannot must
Salesforce is highly configurable — almost nothing is universally true across all orgs, editions, and objects.
Ask: does this require setup or config to be true? If yes → the absolute statement is likely wrong.
Example: "Every day" for dashboard refresh is wrong — only true if a schedule is configured.
3
Unknown Features
Decode the Feature Name
Salesforce names features descriptively. Break the name into components and reason about each word.
Row-level formula → formula per row. Grant Access Using Hierarchies → grants access through hierarchy.
Ask: who uses it? what does it do? where does it live? when does it fire?
Even partial decoding gives enough context to evaluate whether a feature fits the scenario.
4
Design Principles
Use Salesforce's Philosophy
Least Privilege: defaults are restrictive. Automatic broad access without config = wrong answer.
Scalability: one asset with filters beats many duplicate copies. Always prefer the elegant, maintainable option.
Clicks not Code: for admin questions, declarative (Flow, rules) beats code-based solutions.
Separation of Concerns: security tools handle access, automation handles process, reports handle visibility.
5
Constraints
Read Constraints Before Options
Before reading options, extract every requirement (what must happen) and constraint (what's not allowed).
Any answer violating even one constraint is eliminated — even if it perfectly satisfies the requirement.
Common signals: without modifying only using without admin help in a single
Example: "without modifying the object" immediately eliminates adding a formula field — even though it technically works.
6
Adjacent Reasoning
Cross-Reference Known Facts
Even without the specific fact, adjacent knowledge constrains what the answer can be.
Chain known facts: reports show live data + dashboards built on reports + no auto behavior without config = no auto daily refresh.
Partial knowledge + logical chaining often reaches the right answer even on unknown topics.
Grant Access Using Hierarchies is always ON for standard objects — eliminate any answer that says to enable it on standard objects.
7
Tiebreaker
Most Targeted Solution Wins
When 2 options both seem correct, pick the one that solves only the stated problem without unnecessary side effects.
Report-level solution (row-level formula) beats object-level solution (formula field) when the need is report-specific.
Temporary fix beats permanent change when temporary is sufficient.
One dashboard with filters beats four separate dashboards — Salesforce always favors scalable, maintainable solutions.
8
Process
Flag & Return — Don't Spiral
If stuck after ~60 seconds: make your best elimination-based guess, flag the question, move on immediately.
Later questions often activate related knowledge that makes flagged questions suddenly clear.
Time lost on one hard question = easy questions you haven't reached yet.
Up to 5 questions are unscored — don't spiral on something that may not even count.
⚠
Trap Type
The Partial Knowledge Trap
The most dangerous wrong answers contain a real number or real feature from the right context — applied incorrectly.
Example: 20 (charts & tables sub-limit) vs. 25 (total widget limit). Both are real — one is the total, one is a sub-limit.
When you recognize a number, pause: is this the total or a sub-limit? the feature itself or a setting within it?
Feeling of recognition ≠ correctness. The trap is designed to feel right.
Quick Decision Frameworks
Framework · Data Management
Which Import Tool?
①
Volume over 50K records? → Data Loader only. Wizard eliminated immediately.
②
Non-admin user importing Contacts/Leads? → Basic Data Import (enabled in Setup + profile permission).
③
Complex upsert, custom external ID, or Record Type ID needed? → Data Loader.
④
Standard objects, under 50K, admin doing it? → Data Import Wizard.
Framework · Security
Which Access/Sharing Tool?
①
Can't see an object at all? → Check Profile (object permissions).
②
Can see object but not others' records? → Check OWD + Role Hierarchy.
③
Can see record but a field is missing? → Check Field-Level Security in Profile.
④
Need horizontal sharing across teams? → Sharing Rules (not Role Hierarchy).
Framework · Automation
Which Flow Type?
①
Record saved/changed triggers it? → Record-Triggered Flow.
②
Runs on a time schedule? → Scheduled-Triggered Flow.
③
User sees screens and provides input? → Screen Flow.
④
Only updating fields on triggering record? → Before Save. Anything else (creating records, emails, other objects)? → After Save.
Framework · Object Manager
Lookup vs. Master-Detail?
①
Child can exist without parent? → Lookup relationship.
②
Child meaningless without parent / cascade delete OK? → Master-Detail.
③
Need roll-up summary fields on parent? → Must use Master-Detail.
④
Many-to-many needed? → Junction object with 2 Master-Detail fields. Max 2 M-D fields per custom object.
Framework · Analytics
Report Filter Scopes
👤
My Records — only records you personally own.
👥
My Team's Records — you + everyone below you in the Role Hierarchy.
🌐
All Records — every record your sharing model allows you to see.
⚠️
Dashboard filters are NOT security — users can switch filters freely. Real security = sharing model.
Framework · Data Management
Deletion Recoverability
🟢
Records — 15-day Recycle Bin. Most recoverable.
🟡
Fields — 15-day field recycle bin. Data permanently gone after window.
🔴
Objects — essentially permanent. No standard recovery path. Most dangerous.
⚫
Users — cannot be deleted. Deactivate instead. Preserves data integrity and record history.

Topic Area: Configuration & Setup (15%)
Config & Setup
Company Information & Fiscal Year
Company Information in Setup sets the default locale, language, currency, and time zone for the org.
Fiscal Year settings control how Salesforce calculates quarter-based reports and forecasting. Can be set to standard (calendar) or custom.
Business Hours: define when your support team operates — used by escalation rules and case management.
Holidays can be defined and linked to Business Hours to automatically exclude them from case escalation timers.
Config & Setup
User Management Essentials
Users cannot be deleted — only deactivated. Deactivation frees the license but preserves all record ownership and history.
Each user must have exactly one Profile and one Role (though role is optional).
Freeze a user to immediately block login without deactivating — useful when investigating suspicious activity.
Delegated Administration lets non-admin users manage users in specific roles — useful for large orgs with regional admins.
Config & Setup
Login & Session Security
Session Settings control timeout duration, IP restrictions, and login hours per profile.
Login Hours on a profile restrict when users with that profile can log in — outside those hours, access is blocked.
Login IP Ranges on a profile restrict which IP addresses users can log in from.
Multi-Factor Authentication (MFA): admins can require MFA for all users via Setup — increasingly required by Salesforce.
Topic Area: Object Manager & Lightning App Builder (15%)
Object Manager
Field Types — Key Distinctions
Formula: calculated, read-only, cannot be directly edited by users. References other fields.
Roll-Up Summary: only on Master-Detail parent. Calculates COUNT, SUM, MIN, MAX across child records.
Controlling vs Dependent Picklists: controlling field drives which values appear in dependent field. Both can be picklists or checkboxes can control a picklist.
External ID: marks a field as a unique identifier for upsert operations — used by Data Loader to match records.
Object Manager
Page Layouts vs Record Types
Page Layout: controls which fields, related lists, and buttons appear on a record page. Assigned per profile per record type.
Record Type: drives which picklist values, sales process, and page layout a user sees. Assigned per profile.
A user sees a record type's picklist values based on their profile's record type assignment — not just the page layout.
Record Types must be created before Paths can be built — sales process must be created before record types.
App Builder
Lightning App & Page Concepts
Lightning App Builder creates custom record pages, home pages, and app pages using drag-and-drop components.
Custom record pages can be assigned to specific apps, record types, and profiles for tailored experiences.
Dynamic Forms: move fields from page layouts onto the Lightning record page directly, enabling field visibility rules without multiple layouts.
Utility Bar: persistent footer in Lightning apps — add items like Tasks, History, Notes, Macros for quick access.
Topic Area: Workflow & Process Automation (15%)
Automation
Flow Types — Quick Reference
Record-Triggered: fires on record create/update/delete. Before Save (update triggering record efficiently) or After Save (create/update other records).
Scheduled: runs on a time schedule — queries records and processes a batch. Like a daily alarm clock for data.
Screen Flow: user-facing screens collect input. Embedded in pages, launched from buttons, or added as utility items.
Autolaunched (No Trigger): called by another flow or Apex. No UI, no trigger of its own — purely reusable logic.
Automation
Flow Elements — Know Each One
Get Records: queries Salesforce for records matching criteria. Think: SELECT.
Create / Update / Delete Records: DML operations on records. Always bulkify — collect records in a Loop then update outside the Loop in one operation.
Decision: if/then branching — routes the flow down different paths based on conditions.
Loop: iterates through a collection one record at a time. Assignment inside loop → DML element outside loop.
Automation
Validation Rules & When Conditions Fire
Validation Rules fire during record save — they prevent saving when the formula returns TRUE (not false).
They apply to all save operations including Data Loader updates — temporarily deactivate if bulk updating legacy records that predate the rule.
Order of execution: Validation Rules fire before Before Save Flows but after system validation (required fields, data type checks).
ISBLANK() vs ISNULL(): ISBLANK checks both null and empty string — use ISBLANK for text fields. ISNULL works for non-text fields.
Topic Area: Sales & Marketing Applications (10%)
Sales & Marketing
Lead Management
Lead Conversion: creates an Account, Contact, and optionally an Opportunity from a Lead. Original Lead record is then marked converted.
Lead Assignment Rules: automatically assign incoming leads to users or queues based on criteria. Only one rule can be active at a time.
Web-to-Lead: generates an HTML form that captures lead info from a website directly into Salesforce. Max 500 leads per day.
Lead Source field tracks where a lead originated — carried over to the Opportunity on conversion for reporting.
Sales & Marketing
Opportunities & Quotes
Sales Process: defines which Opportunity stages are available. Assigned to a Record Type. Must be created before Record Types.
Quotes: generated from an Opportunity — captures products, pricing, and terms to send to a customer. Must be enabled in Setup.
Price Books: Standard Price Book exists by default. Custom Price Books allow different pricing for different customer segments.
Syncing a Quote with an Opportunity locks the opportunity's line items to the quote — only one quote can be synced at a time.
Sales & Marketing
Campaigns & Forecasting
Campaigns: track marketing efforts and their ROI. Leads and Contacts can be Campaign Members with a member status (Sent, Responded, etc.).
Campaign Influence links Opportunities to Campaigns that contributed to winning them — important for marketing ROI reporting.
Forecasting: tracks expected revenue from opportunities. Forecast categories (Pipeline, Best Case, Commit, Closed) roll up to manager level via Role Hierarchy.
Paths: visual stage guidance on Opportunities — key fields and success guidance appear at each stage. Must enable Path in Setup first.
Topic Area: Service & Support Applications (10%)
Service & Support
Case Management Rules
Assignment Rules: automatically route cases to users or queues based on criteria. Only one active rule per object.
Auto-Response Rules: automatically send an email to the customer when a case is created. References an email template. One active rule per object.
Escalation Rules: automatically escalate cases that haven't been resolved within a time threshold. Respects Business Hours settings.
All three rule types require Email-to-Case to be enabled for inbound email scenarios. Email-to-Case must be enabled in Setup first.
Service & Support
Queues, Entitlements & Channels
Queues: shared holding areas for cases (or leads, custom objects). Members of the queue can take ownership of items.
Entitlements: define the level of support a customer is entitled to — links to Accounts and Contacts, drives SLA tracking.
Email-to-Case: converts inbound emails to case records automatically. Must be enabled in Setup. On-Demand Email-to-Case works without firewall changes.
Web-to-Case: captures cases from a website form. Max 5,000 cases per day. Similar to Web-to-Lead but for cases.
Service & Support
Knowledge & Macros
Knowledge Articles: structured documentation agents can reference and share with customers. Articles have types and data categories.
Knowledge articles go through a publication lifecycle: Draft → In Review → Published → Archived. Only published articles are visible to customers.
Macros: automate repetitive service tasks — a macro can send an email, update fields, and log an activity in one click. Added to the Utility Bar.
Quick Text: pre-written message snippets agents can insert into emails, chats, and feeds — saves time on common responses.
Topic Area: Data & Analytics Management (17% — Highest Weight)
Data & Analytics
Report Types — Know All Four
Tabular: flat list of records, no grouping. Cannot be used as dashboard source (no summarized data). Good for exports.
Summary: groups records by a field, shows subtotals. Can be used as dashboard source. Most commonly tested report type.
Matrix: groups by rows AND columns — shows a grid of summarized data. Good for comparing across two dimensions.
Joined: combines up to 5 report blocks with different report types into one view. Most complex. Cannot be used as dashboard source.
Data & Analytics
Report Features — Row-Level Formula, Buckets, Cross-Filters
Row-Level Formula: adds a calculated column inside the report. Doesn't modify the object. Can calculate date differences between fields. Available in Lightning Report Builder.
Bucket Column: categorizes existing field values into named groups within a report. Cannot calculate date differences — categorization only.
Cross-Filters: filter report records based on whether related records exist or don't exist. Example: accounts WITH no open opportunities.
Custom Report Types: define which objects and fields are available to a report — needed when standard types don't cover the required objects/relationships.
Data & Analytics
Dashboards — Key Concepts
Dashboards are built from reports. Source report must exist before the dashboard component can be created.
Running User: determines whose data lens the dashboard uses. All viewers see data through that user's sharing model.
Dynamic Dashboard: each viewer sees data through their own sharing model — no fixed running user. Limit of 3, 5, or 10 dynamic dashboards depending on edition.
Refresh options: manual (click Refresh), scheduled (daily/weekly/monthly). Not automatic by default. Tabular reports cannot be used as dashboard source.
Data & Analytics
Data Backup & Export
Data Export Service: Setup tool that exports all org data as CSV files. Can be scheduled weekly or monthly. Manual export available anytime.
Reporting Snapshots: scheduled capture of summary/matrix report results into a custom object — enables historical trend reporting. Must be set up proactively — cannot reconstruct past data retroactively.
Mass Delete: bulk delete records meeting criteria. Sends to Recycle Bin (15-day window). Max 50,000 records per delete operation via UI.
Field History Tracking retains for 18 months — provides record-level changelog but NOT aggregate historical snapshots.
Topic Area: Productivity & Collaboration (10%)
Productivity
Activities — Tasks & Events
Tasks: to-do items with a due date, status, and priority. Can be assigned to another user. Linked to records.
Events: time-based meetings with a start and end time. Appear on the calendar. Can have invitees.
Activity Timeline: shows all past and upcoming activities on a record in chronological order. Must be enabled in Activity Settings.
Recurring Tasks and Events: can be created on a schedule — daily, weekly, monthly. Editing one in a series can affect all or just that one.
Productivity
Chatter & Collaboration
Chatter: Salesforce's internal social network. Users can post on records, groups, and profiles. @mentions notify specific users.
Chatter Groups: public (anyone can join and see), private (join by request, content hidden), unlisted (invitation only, hidden from search).
Chatter Free users: external users given Chatter access only — no Salesforce license required. Cannot see CRM data.
Following: users can follow records to receive Chatter notifications when the record is updated or commented on.
Productivity
Email Integration & Templates
Email Templates: stored in folders (folder-based sharing). Classic (HTML, text, custom) or Lightning (enhanced). Apply letterhead for branding.
Letterheads: define the branded header and footer applied to HTML email templates — create once, apply to many templates.
Einstein Activity Capture: syncs emails and events between Salesforce and Gmail/Outlook. Activities are captured but may not appear in standard Activity reports.
Mass Email: send one email to multiple contacts or leads at once using a template. Subject to org daily email limits.
Topic Area: Agentforce AI (8% — New in 2025)
Agentforce AI
Prompt Builder — Core Concepts
Prompt Builder creates reusable AI prompt templates that generate contextual text from live Salesforce record data using merge fields.
Template types include Record Summary (summarizes a record), Sales Email (drafts outreach), and others depending on org configuration.
Templates must be activated before they can be used. They are accessed from the relevant record page via Einstein Panel or a custom button.
Prompt Builder requires a specific Developer Org or AI-enabled edition — not available in all standard orgs.
Agentforce AI
Agentforce Agents — Concepts & Security
Agentforce Agents are AI-powered assistants configured to handle specific topics — case triage, order questions, policy lookups — autonomously.
Agents are built in Agent Builder. You define topics (what the agent handles), instructions (how it handles them), and actions (what it can do).
AI Security: agents operate within the sharing model of the configured user — they can only access records that user can see. AI does not bypass security.
Troubleshooting agent permissions: check the agent's running user profile and sharing model — same diagnostic approach as any user access issue.
Agentforce AI
Einstein Features Admins Should Know
Einstein Lead Scoring: uses AI to score leads based on likelihood to convert. Based on historical data patterns in your org.
Einstein Opportunity Scoring: predicts likelihood of winning an opportunity. Scores are visible on opportunity records and list views.
Einstein Activity Capture: automatically logs emails and calendar events against Salesforce records by syncing with Gmail or Outlook.
AI features generally require sufficient data to train on — orgs with minimal historical data may see less accurate predictions.
Security Model — Consolidated Reference
Security Model · Full Picture
The Three-Axis Security Model — Profiles, Roles, Sharing
Axis 1 — What Can You Do? (Profiles & Permission Sets)
Object Permissions: Read, Create, Edit, Delete, View All, Modify All — per object per profile.
Field-Level Security: visible or hidden, editable or read-only — per field per profile.
System Permissions: export data, manage users, customize app — platform-level capabilities.
Permission Sets add permissions. They never restrict. One profile + many permission sets per user.
Axis 2 — Which Records Can You See? (OWD + Role Hierarchy + Sharing)
OWD: most restrictive baseline. Private = owner + above in hierarchy only.
Role Hierarchy: opens visibility upward. Manager sees all records owned by anyone below them.
Sharing Rules: horizontal sharing — criteria or ownership based. Opens access across hierarchy branches.
Manual Share: record owner grants specific user access to a specific record. One-off, ad hoc.
Axis 3 — Which Fields Within Records? (FLS in Profile)
Even with record access, individual fields can be hidden via Field-Level Security.
FLS is enforced everywhere — page layouts, reports, API — not just the record page.
Both gates must be open: record-level access AND field-level visibility needed to see a field.
Grant Access Using Hierarchies: always ON for standard objects. Configurable (can be disabled) only on custom objects.
